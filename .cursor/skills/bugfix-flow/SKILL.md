# Bugfix Flow

## 목적

버그를 단순히 빠르게 고치는 데서 끝내지 않고, 원인과 회귀 위험까지 함께 관리한다.

## 사용 시점

아래 상황에서 우선 적용한다.

- 오류 수정
- 화면 이상 동작 수정
- API 예외 수정
- 데이터 처리 오류 수정
- 스타일 깨짐/상태 처리 버그 수정

## 기본 원칙

일반적인 실패·재시도·「모름」 표기는 `shared/rules/working-principles.mdc` **조사·소통·실패 대응**을 따른다. 본 스킬은 **버그**에 한해 재현→원인→최소 수정 순서를 구체화한다.

- 가장 작은 수정으로 원인을 해결한다.
- 추정 수정을 사실처럼 단정하지 않는다.
- 구조 변경이 필요하면 버그 수정 범위를 재정의한다.
- 수정 후 영향 범위와 부작용 가능성을 반드시 점검한다.

## 실행 절차

### 1) 증상과 기대 동작 정리

- 현재 증상을 관찰 가능한 형태로 기록한다.
- 기대 동작을 명확히 정의한다.
- 입력/환경/사용자 행동 기준으로 증상과 기대를 분리해 적는다.

출력 예시:
- 현재 증상: [무엇이 어떻게 잘못되는지]
- 기대 동작: [원래 어떻게 동작해야 하는지]

### 2) 재현 조건과 영향 범위 확인

- 재현 절차(선행 조건, 입력값, 환경)를 확인한다.
- 항상 발생/간헐 발생 여부를 구분한다.
- 영향 범위를 기능, 사용자군, 화면/API, 데이터 경로 관점에서 정리한다.

출력 예시:
- 재현 조건: [재현 단계]
- 영향 범위: [영향 받는 기능/사용자/시스템]

### 3) 담당 에이전트 우선 고려

- UI/상호작용/스타일 문제는 `frontend-agent`를 우선 고려한다.
- 서버 로직/API/데이터 무결성 문제는 `backend-agent`를 우선 고려한다.
- 경계가 겹치면 원인 지점을 기준으로 주 담당을 정하고, 필요 시 협업한다.

### 4) 가장 작은 수정으로 원인 해결

- 증상 완화가 아닌 원인 제거를 목표로 수정한다.
- 관련 없는 리팩터링/구조 변경은 분리한다.
- 계약(API/스키마/정책) 변경이 필요하면 즉시 범위를 재정의하고 사용자 확인 후 진행한다.

### 5) 수정 범위와 부작용 가능성 점검

- 변경 파일/모듈/흐름을 기준으로 영향 범위를 다시 점검한다.
- 성공 경로뿐 아니라 실패/예외/상태 처리(로딩/빈값/오류/권한 등)를 함께 확인한다.
- 기존 동작을 깨뜨릴 수 있는 지점(회귀 후보)을 목록화한다.

### 6) `qa-agent`로 회귀 여부 검토

- 수정된 동작과 인접 기능 중심으로 회귀 위험을 검토한다.
- 필요 시 최소 검증 시나리오를 추가한다.
- “무엇을 확인했고 무엇이 남았는지”를 분리해 기록한다.

### 7) `docs-agent`로 원인/수정 내용 정리

- 버그 원인, 수정 방식, 영향 범위, 검증 결과를 간단히 문서화한다.
- 후속 확인 포인트(모니터링, 추가 테스트 필요 항목)를 남긴다.

## 결과물

작업 완료 시 아래 산출물을 반드시 남긴다.

- 원인 설명
- 수정 코드
- 영향 범위 메모
- 회귀 점검 결과

## 예외 처리

### 재현 불가능한 경우

- 추정 수정을 확정 원인처럼 단정하지 않는다.
- 아래를 반드시 기록한다.
  - 원인 후보
  - 추가 확인 포인트
  - 현재까지의 관찰 근거

### 구조 변경이 필요한 경우

- 단순 버그 수정으로 위장하지 않는다.
- 변경 범위를 “구조 개선/설계 변경”으로 재정의하고, 승인 후 진행한다.

## 최소 완료 체크리스트

- [ ] 증상/기대 동작이 분리되어 정리됨
- [ ] 재현 조건과 영향 범위가 기록됨
- [ ] 원인 기반 최소 수정이 반영됨
- [ ] 부작용/상태 처리 점검이 수행됨
- [ ] `qa-agent` 회귀 검토 결과가 기록됨
- [ ] `docs-agent` 요약 문서가 남음
