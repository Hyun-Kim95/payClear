# cross-check

## 목적
하나의 주제·질문을 **서로 다른 모델**에 **병렬**로 던져 독립 답변을 모으고, 메인 에이전트가 **교차검증 종합**(패턴 A)을 한다. 사용자가 원하면 라운드1 답변을 서로에게 보여주고 **반박·보완하는 2라운드 토론**(패턴 B)을 이어간다.

## 언제 쓰나
- 단일 모델 답변의 편향·사각지대를 줄이고 싶을 때
- 의사결정·설계 트레이드오프처럼 **여러 관점**이 필요한 주제
- 슬래시 `/cross-check <주제>` 입력 시

## 사용법
```text
/cross-check <주제 또는 질문>
```
- 검토할 **주제 전문**은 슬래시 **앞이든 뒤든** 어디에 와도 된다. 긴 글을 먼저 붙여넣고 맨 끝에 `/cross-check`만 붙여도 그 전체가 주제다. 위치는 무관하며, 메시지 본문 전체를 주제·맥락으로 읽는다.
- 맥락(배경·제약·목표)이 있으면 함께 적을수록 답변 품질이 올라간다.
- 메시지에 주제 본문 없이 `/cross-check`만 있으면 무엇을 검토할지 1줄로 되묻는다.

## 핵심 메커니즘
- "서로 다른 모델"은 **`Task` 도구의 `model` 파라미터**로 구현한다. 같은 프롬프트를 `model`만 바꿔 **여러 서브에이전트로 병렬 호출**한다.
- 답변자는 의견만 내면 되므로 **`readonly: true`**, **`run_in_background: false`** 로 띄우고, **한 메시지에 여러 `Task` 호출**을 넣어 동시에 실행한다.
- 모델끼리 직접 대화는 불가능하다. 라운드1 답변을 **메인이 모아 라운드2 프롬프트에 넣어** "서로의 답을 보여주는" 토론을 오케스트레이션한다.
- 서브에이전트는 이 대화 맥락을 모른다. **주제·배경을 프롬프트에 전부 임베드**한다.

## 모델 구성 (기본값)
답변자 2개 + 종합자(메인) 1개 = 서로 다른 3개 관점.

| 역할 | 모델 |
|--|--|
| 답변자 1 | `claude-opus-4-8-thinking-high` |
| 답변자 2 | `gpt-5.5-medium` |
| 종합자 | 메인 에이전트(현재 세션 모델) |

- 사용자가 입력에서 모델을 지정하면 그대로 쓴다.
- 지정 모델이 **현재 환경에서 사용 불가**면 임의로 바꾸지 말고 사용 가능한 슬러그를 알리고 확인을 받는다.
- 종합자(메인)가 답변자와 같은 계열이어도 무방하나, 가능하면 답변자에 서로 다른 벤더를 섞어 다양성을 확보한다.

## 패턴 A — 독립 수집 + 교차검증 (라운드 1)

### 절차
1. 주제를 확정한다(필요 시 1줄 확인).
2. **한 메시지에서** 답변자 수만큼 `Task`를 **병렬** 호출한다. 각 호출은 `model`만 다르고 **프롬프트는 동일**, `subagent_type: "generalPurpose"`, `readonly: true`, `run_in_background: false`.
3. 각 서브에이전트의 독립 답변을 수집한다(서로의 존재를 모른 상태).
4. 메인이 교차검증해 종합한다.

### 답변자 프롬프트 템플릿 (모든 답변자 동일)
```text
아래 주제에 대해 너 스스로의 판단으로 독립적인 답을 작성하라.
다른 모델의 답은 주어지지 않는다. 추측이면 추측이라고 표시하라.

[주제]
<주제 전문 + 배경/제약/목표>

[출력 형식]
- 핵심 결론: (2~3문장)
- 근거: (불릿)
- 핵심 가정·전제:
- 리스크·반례·실패 케이스:
- 불확실/모르는 부분:
```

### 종합 출력 형식 (라운드 1)
```markdown
## 라운드 1 — 독립 답변 교차검증

### 모델별 핵심 결론
| 모델 | 핵심 결론 |
|--|--|
| <답변자1> | … |
| <답변자2> | … |

### 합의된 지점
- …

### 불일치/충돌 지점
- 쟁점: <한 줄> — <답변자1 입장> vs <답변자2 입장>

### 각자 놓친 점 / 보완 필요
- …

### 종합 권고 (메인 판단)
- … (불확실하면 불확실로 표시)
```

### 라운드1 종료 후 (HUMAN)
종합을 제시한 뒤 **2라운드(패턴 B) 진행 여부를 묻는다.** 사용자가 진행을 원하면 패턴 B로 넘어간다. 자동으로 2라운드를 돌리지 않는다.

(예외) 사용자가 처음부터 "2라운드까지", "토론까지", "패턴 B" 등을 명시했으면 묻지 않고 패턴 B까지 진행한다.

## 패턴 B — 반박·보완 토론 (라운드 2)

라운드1 답변을 입력으로 삼아, 각 모델에게 **다른 모델의 답을 보여주고** 반박·보완하게 한다.

### 절차
1. 라운드1의 **각 답변 원문**을 모델명과 함께 정리한다.
2. **한 메시지에서** 답변자 수만큼 `Task`를 다시 **병렬** 호출한다(라운드1과 동일 설정). 각 모델에는 **자신의 답 + 다른 모델들의 답**을 함께 준다.
3. 각 모델의 2라운드 답변(유지/수정/반박)을 수집한다.
4. 메인이 최종 종합한다.

### 답변자 프롬프트 템플릿 (라운드 2)
```text
같은 주제에 대해 여러 모델이 1라운드 답을 냈다. 아래를 읽고
(1) 다른 답의 약점·오류·빠진 점을 구체적으로 반박하고,
(2) 타당한 지적은 받아들여 네 입장을 보완하라.
바뀐 점이 있으면 "수정", 유지하면 "유지"로 명시하라.

[주제]
<주제 전문>

[네 1라운드 답]
<해당 모델의 라운드1 답>

[다른 모델들의 1라운드 답]
- <답변자X>: <원문>
- …

[출력 형식]
- 다른 답에 대한 반박: (대상 + 근거)
- 내 입장 보완/수정 (수정/유지 명시):
- 최종 결론:
- 남은 이견(합의 불가 지점):
```

### 종합 출력 형식 (라운드 2)
```markdown
## 라운드 2 — 반박·보완 결과

### 입장 변화
| 모델 | 1라운드 | 2라운드 | 변화 |
|--|--|--|--|
| <답변자1> | … | … | 유지/수정 |

### 수렴된 합의
- …

### 끝까지 남은 이견
- 쟁점: … — 각 입장과 그 근거

### 최종 권고 (메인 판단)
- 채택안 + 이유, 남은 리스크/조건
```

## 주의 / 리스크
- **비용·시간**: 답변자 N개 × 라운드 수만큼 모델 호출이 곱으로 늘어난다(2모델×2라운드 = 4회 + 종합). 라운드2는 사용자가 원할 때만.
- **readonly 제약**: 답변자 서브에이전트는 인터넷·MCP 접근이 없다. 최신 웹정보가 핵심인 주제는 메인이 별도로 보강하거나 사용자에게 알린다.
- **동률/합의 불가**: 모델 의견이 갈리면 억지로 하나로 합치지 말고 **이견을 그대로 보존**하고 메인 판단을 별도 표시한다.
- **맥락 누락**: 서브에이전트는 대화 맥락을 모르므로 주제·제약을 프롬프트에 빠짐없이 넣는다.
- **모델 가용성**: 지정 모델이 없으면 임의 대체하지 말고 확인한다.
