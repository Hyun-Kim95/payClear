# Document Change

## 목적

구현 결과를 사용자/팀이 바로 이해하고 다음 행동을 결정할 수 있도록 문서화한다.

## 사용 시점

- 작업 완료 후 결과 공유가 필요할 때
- `README` 또는 운영 문서를 업데이트할 때
- 인수인계 문서를 작성하거나 보강할 때
- 변경 이력(릴리즈 노트, 작업 로그)을 정리할 때
- 병렬 구현 중 API 계약/스키마/오류 규칙이 변경되어 즉시 동기화가 필요할 때

## 기본 원칙

- 사실(무엇)과 의도(왜)를 분리해 쓴다.
- 현재 구현 기준으로 작성하고, 미확정 사항은 명시한다.
- 영향 범위를 코드/문서/운영 관점으로 나눠 적는다.
- 변경이 사용자 행동에 미치는 효과를 먼저 설명한다.
- 계약(API, 스키마, 오류 규칙) 변경은 가장 먼저 공유한다.

## 작업 절차

### 0) 요구·문서·구현 삼각 점검

PRD·Gate·API·스킬/규칙·README를 건드린 변경은 **요구 ↔ 문서 ↔ 구현**이 맞는지 spot check한다. 「PRD 빠짐」「문서 반영」「범위 아니었나」 재지적이 나오기 쉬운 구간이다.

### 1) 병렬 작업 동기화 선확인

병렬 작업이 진행 중이면 아래를 먼저 확인한다.

- 최신 API 계약(엔드포인트, 요청/응답, 상태 코드, 오류 포맷)
- UI 계약(입력/출력, 상태 모델, 예외 표시 방식)
- 문서와 실제 구현의 일치 여부

불일치가 있으면 문서 초안 작성 전에 차이를 먼저 기록한다.

### 2) 무엇이 바뀌었는지 정리

- 기능 추가/수정/삭제 항목
- 동작 변화(사용자 관점 + 시스템 관점)
- 상태 처리 변화(기본/로딩/빈/오류/권한 등)

### 3) 왜 바꿨는지 정리

- 변경 배경(버그 수정, 정책 반영, 성능, 유지보수성)
- 기대 효과(문제 감소, 사용성 개선, 운영 안정성)
- 트레이드오프 또는 제한사항

### 4) 영향 파일/영역 정리

- 코드: 모듈/컴포넌트/서비스/DB/설정
- 문서: `README`, 가이드, API 명세, 런북
- 운영: 배포 변수, 마이그레이션, 모니터링/알람

가능하면 경로 단위로 명시한다.

### 5) 추가 확인 필요 항목 표시

- 미검증 케이스, 잠재 회귀 위험
- 외부 의존성/권한/환경 변수 확인 필요 사항
- 사용자 승인 또는 타 팀 확인이 필요한 결정

### 6) 필요 시 docs-agent로 문서 다듬기

문서 톤, 구조, 누락 보완이 필요하면 `docs-agent`에 전달해 가독성과 전달력을 개선한다.

## 출력 템플릿

아래 템플릿을 기본으로 사용한다.

```markdown
## 변경 요약
- [핵심 변경 1]
- [핵심 변경 2]

## 영향 범위
- 코드: [영향 모듈/파일/영역]
- 문서: [업데이트된 문서]
- 운영/배포: [환경 변수, 마이그레이션, 모니터링 등]

## 사용 방법 또는 확인 포인트
- [사용자/팀이 바로 확인할 항목]
- [검증 방법 또는 재현 절차]

## 후속 작업 메모
- [추가 작업]
- [담당/의존 관계/우선순위]
```

## 빠른 체크리스트

- [ ] 병렬 작업 시 최신 API/UI 계약과 문서가 일치한다.
- [ ] 무엇이 바뀌었는지와 왜 바꿨는지가 분리되어 있다.
- [ ] 영향 범위가 코드/문서/운영 관점으로 정리되어 있다.
- [ ] 사용 방법 또는 확인 포인트가 실행 가능하게 작성되어 있다.
- [ ] 후속 작업 메모에 미확정/리스크가 표시되어 있다.
- [ ] 필요 시 `docs-agent`로 문서 완성도를 보강했다.
